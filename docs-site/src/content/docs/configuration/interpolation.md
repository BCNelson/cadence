title: Interpolation
description: ${env:VAR}, ${env:VAR:-default}, ${file:...}, $$ — when they run and how.

cadence supports two interpolation schemes in string values: `${env:...}` for reading environment variables, and `${file:...}` for reading file contents. Both run **after** layer merge, so an overlay can substitute a token from an underlying layer.

## Phases and order

Interpolation runs in two passes, in this order:

1. **`${env:VAR}`** or **`${env:VAR:-default}`** — replaced by the environment variable's value.
2. **`${file:PATH}`** — replaced by the contents of the file (with trailing whitespace and newlines trimmed). Relative paths resolve against the directory of the YAML file that introduced the token.

Env first means a `${file:...}` token's argument can itself be an `${env:...}` token, so an operator can parameterize the secret path via systemd `EnvironmentFile=` (or any other env source) without baking the path into YAML:

```yaml
server:
  uuid_salt: "${file:${env:CADENCE_UUID_SALT_FILE}}"
```

The reverse — a file whose contents contain `${env:...}` tokens — is **not** re-expanded; the file pass is the last substitution phase.

## Forms

```yaml
server:
  uuid_salt: "${env:CADENCE_UUID_SALT}"                          # required env var
  base_url:  "${env:CADENCE_BASE_URL:-http://localhost:8080}"    # with fallback

channels:
  - name: ops
    type: webhook
    url: "${file:/run/secrets/ops-webhook.url}"                  # read URL from secret file
    headers:
      Authorization: "Bearer ${env:OPS_TOKEN}"

ping_keys:
  - name: cron
    key: "${file:${env:CADENCE_CRON_KEY_FILE}}"                  # path comes from env, contents from file
```

## Using `${file:...}` with secret managers

`${file:...}` is the bridge to any secret manager that materializes secrets as files on disk — agenix (`age.secrets.<name>.path`), sops-nix (`sops.secrets.<name>.path`), Kubernetes Secrets mounted as files, Docker secrets at `/run/secrets/*`, or systemd's `LoadCredential=`. The token itself is non-secret, so it's safe to commit alongside the rest of the YAML.

The secret file must be readable by the cadence user/UID at startup; trailing newlines are trimmed, so files written by these managers work without massaging. See the [NixOS install page](/cadence/install/nixos/#secrets) for the agenix/sops-nix-specific wiring.

## Errors and edge cases

- **`${env:VAR}` unset, no default** — fails startup with a precise file/line reference.
- **`${env:VAR:-default}` unset or empty** — falls back to `default` (literal string, no nesting).
- **`${file:path}` missing or unreadable** — fails startup with the resolved absolute path in the message.
- **`$$`** — a literal `$`. Two dollar signs in a YAML string survive interpolation as one `$` in the resolved string. Use this when you legitimately need a `$` next to a `{`.
- **Unrecognized scheme** — `${other:...}` is left untouched after env/file passes. Any `${...}` surviving the second pass is an error.

## Where it applies

Interpolation runs on every scalar string value after merge. It does not run on YAML map keys, on numbers, or on booleans — only string values.

## Security note

The settings file generated by the [NixOS module](/cadence/install/nixos/) lives in the world-readable Nix store. Always use `${env:...}` with an `environmentFile`, `${file:...}` with a path from a secret manager, or `extraConfigFiles` for the secret-bearing YAML — never inline the secret.
